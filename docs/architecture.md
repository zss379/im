## 1. 概述

### 1.1 文档目的

本文档描述数莲 PaaS 平台 IM 系统的技术架构设计，涵盖系统分层架构、组件职责、关键业务流程时序、部署拓扑及技术选型决策，作为开发团队实现落地与技术评审的依据。

### 1.2 架构设计原则

| 原则 | 说明 |
|------|------|
| **高可用** | 无单点故障，微服务多副本部署，年度可用性 ≥ 99.9% |
| **高并发** | 支持 10 万并发在线，单聊 5000 TPS，P99 消息延迟 < 200ms |
| **水平扩展** | MongoDB 分片 + 微服务无状态化，按压力线性扩容 |
| **数据安全** | 全链路 TLS 1.3 加密，租户数据行级隔离，敏感字段加密存储 |
| **可观测** | 全链路日志、Prometheus 指标、Loki 日志聚合、结构化告警 |

### 2.2 层职责定义

| 层级 | 职责 | 关键组件 |
|------|------|---------|
| **客户端层** | UI 渲染、WebSocket 连接管理、本地缓存、消息发送/接收 | OpenIM SDK、本地 DB (SQLite/WatermelonDB) |
| **接入层** | SSL 终结、负载均衡、WebSocket 代理、限流、认证校验 | Nginx、Kong/K8s Ingress |
| **服务层** | 业务逻辑（用户/会话/消息/群组/机器人/风控/审计/文件/卡片） | Go 微服务、OpenIM Server |
| **数据层** | 数据持久化、缓存、消息队列、对象存储 | MySQL、MongoDB、Redis、Kafka、MinIO |
| **可观测层** | 指标采集、日志聚合、链路追踪、告警 | Prometheus、Grafana、Loki、Jaeger |

### 2.3 微服务拆分

| 服务名 | 职责 | 与 OpenIM 关系 | 存储依赖 |
|-------|------|---------------|---------|
| **user-svc** | 用户认证、Token 管理、在线状态、租户切换 | OpenIM 用户体系对接 | MySQL + Redis |
| **conversation-svc** | 会话列表、置顶、免打扰、未读计数 | OpenIM 会话管理封装 | MySQL + Redis |
| **message-svc** | 消息发送、撤回、转发、搜索、SSE | 对接 OpenIM 消息通道 | MongoDB + Kafka |
| **group-svc** | 群 CRUD、成员管理、角色权限、入群申请 | OpenIM 群组管理封装 | MySQL |
| **contact-svc** | 组织架构、成员搜索、名片 | 独立同步 HR 系统 | MySQL |
| **bot-svc** | 机器人配置、Webhook 收发、群机器人 | 调用 message-svc 发送 | MySQL |
| **rc-svc** (risk-control) | 敏感词匹配、频控校验、文件限制 | 消息发送前同步调用 | MySQL + Redis |
| **file-svc** | 文件上传/下载、断点续传、缩略图 | 对接 MinIO/S3 | MySQL + MinIO |
| **audit-svc** | 管理员操作日志、消息审计 | 全服务异步调用 | MongoDB |
| **card-svc** | 卡片模板管理、卡片渲染、外部推送 | 调用 message-svc 发送 | MySQL |
| **openim-server** | WebSocket 长连接、消息路由、多端同步、离线推送 | - | 依赖自身 DB |

## 4. 关键技术决策

### 4.1 架构模式

| 决策 | 选择 | 理由 |
|------|------|------|
| 通信模式 | 同步 HTTP (REST) + 异步 Kafka + 实时 WebSocket | REST 适合查询类、Kafka 适合解耦异步任务、WebSocket 适合实时推送 |
| 消息发送流程 | 同步链式校验 → 异步持久化 → Kafka 推送 | 发送方需要即时反馈，接收方可容忍毫秒级延迟 |
| 状态管理 | Redis 实时 + MySQL 持久化 | 在线状态/未读数高频更新走 Redis，定期落盘 MySQL |
| 全文检索 | MongoDB Text Index | 避免引入 ES，降低运维复杂度（初期够用） |
| 卡片渲染 | 客户端 JSON 模板渲染 | 服务端返回结构化数据，客户端渲染 UI，跨端一致 |

### 4.2 消息可靠投递机制

```
发送方                                        IM Server                   接收方
  │                                             │                         │
  │ 1. POST /messages (client_msg_id)           │                         │
  │─────────────────────────────────────────────>│                         │
  │                                             │                         │
  │ 2. 校验幂等 (client_msg_id 已存在？)          │                         │
  │                                             │                         │
  │ 3. 发送前链: 敏感词 → 频控 → 权限             │                         │
  │                                             │                         │
  │ 4. 写入 MongoDB (status=发送中)              │                         │
  │                                             │                         │
  │ 5. 返回 msg_id                              │                         │
  │<─────────────────────────────────────────────│                         │
  │                                             │                         │
  │ 6. Kafka 推送消息到接收方 topic              │                         │
  │                                             │                         │
  │                         7. WebSocket push   │                         │
  │                         (message:new)       │                         │
  │                         ──────────────────────────────────────────────>│
  │                                             │                         │
  │                         8. ACK (确认接收)     │                         │
  │                         <──────────────────────────────────────────────│
  │                                             │                         │
  │ 9. 更新 status=已发送                        │                         │
  │                                             │                         │
  │ 10. 更新会话 latest_msg + unread_count      │                         │
```

**可靠性保证：**
| 保证 | 机制 |
|------|------|
| 不丢 | Kafka 持久化 + ACK 确认机制 + 离线消息存储 |
| 不重 | client_msg_id 幂等去重（Redis + DB 双重校验） |
| 有序 | 同一会话消息按 send_time 排序，MongoDB 索引保证 |
| 可达 | 离线消息暂存 MongoDB → 上线全量拉取 |

### 4.3 多端同步策略

```
                    ┌─────────────┐
                    │  IM Server  │
                    └──────┬──────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
          ▼                ▼                ▼
     ┌─────────┐     ┌─────────┐     ┌─────────┐
     │  iOS     │     │ Android  │     │  Web    │
     │ 在线     │     │ 在线     │     │ 在线    │
     └─────────┘     └─────────┘     └─────────┘
```

| 同步维度 | 同步方式 | 延迟 |
|---------|---------|------|
| 消息同步 | 发送 → Kafka → 所有在线端 WebSocket push | < 200ms |
| 已读同步 | 端 A 读会话 → API → 端 B 未读清零 | < 500ms |
| 状态同步 | 端 A 切换状态 → API → 所有端广播 | < 1s |
| 置顶/免打扰 | 操作端 → API → 其他端 WebSocket 通知 | < 1s |

### 4.4 离线消息推送策略

```
消息到达 → 检查接收方在线状态
  ├── 在线 → WebSocket 实时推送
  └── 离线 → 判断推送优先级
               ├── 普通消息 → 免打扰抑制
               ├── @我 → 强制推送
               ├── @所有人 → 强制推送
               ├── 审批/系统通知 → 强制推送
               └── 撤回消息 → 不推送
```

**推送渠道：**
| 端 | 在线推送 | 离线推送 |
|----|---------|---------|
| iOS | WebSocket | APNs (通过 JPush 代理) |
| Android | WebSocket | JPush |
| Web | WebSocket | 浏览器 Notification API |

## 9. 安全设计

### 9.1 传输安全

```
客户端 ─── TLS 1.3 ───→ Nginx ─── mTLS (Istio) ───→ 微服务 ─── TLS ───→ 数据库
```

| 层级 | 安全措施 |
|------|---------|
| 客户端 ↔ 网关 | TLS 1.3，移动端证书固定（Certificate Pinning） |
| 网关 ↔ 服务 | Istio mTLS，内部 mTLS 双向认证 |
| 服务 ↔ 数据库 | TLS 连接，数据库账号最小权限 |
| Kafka | TLS + SASL/SCRAM 认证 |

### 9.2 存储安全

| 数据类型 | 加密方式 |
|---------|---------|
| 用户密码 | bcrypt 加盐哈希（cost=10） |
| 手机号 | AES-256-CBC 加密存储，接口返回脱敏 |
| 消息内容 | MongoDB 不加密（TLS 链路保护），磁盘可开启 TDE |
| 日志 | 明文存储（仅内部访问），审计日志不可删改 |
| Token | JWT（RS256 签名），Redis 黑名单机制 |

### 9.3 应用层安全

| 防护 | 措施 |
|------|------|
| RBAC | 群主 > 管理员 > 成员，严格接口权限校验 |
| 租户隔离 | 所有 SQL 查询带 WHERE tenant_id = ? |
| 防 SQL 注入 | GORM 参数化查询，禁止拼接 SQL |
| 防 XSS | 消息文本 HTML 转义渲染 |
| 防 CSRF | Token 验证，Referer 检查 |
| 频率限制 | 全局 100 req/min/user + 消息频控 |
| 输入校验 | 所有用户输入长度/格式校验 |

| `im_rc_rate_limit_total` | Counter | 频控拦截次数 |
| `im_audit_log_write_total` | Counter | 审计日志写入量 |

### 10.3 监控大盘

| 面板 | 内容 |

## 11. 性能优化策略

### 11.1 数据库优化

| 优化手段 | 目标 | 预期效果 |
|---------|------|---------|
| 消息分片 (conversation_id hashed) | MongoDB | 消除单节点瓶颈，均匀写入 |
| 游标翻页 (避免 offset) | 消息拉取 | 深度翻页 O(1) 性能 |
| Redis 缓存未读数 | 未读计数 | 减少 MySQL 写入 90% |
| 敏感词本地 DFA 缓存 | 敏感词匹配 | 匹配耗时 < 100μs |
| MySQL 连接池 | 数据库连接 | 避免频繁建连，max 200 |

### 11.2 缓存策略

| 缓存 | Key 模式 | TTL | 更新策略 |
|------|---------|-----|---------|
| 用户 Token | `token:{token}` | 7天 | 每次请求自动续期 |
| 用户信息 | `user:{id}` | 1h | 变更后删除 |
| 在线状态 | `user:online:{id}` | 120s | 心跳续期 |
| 敏感词库 | 本地内存 | 热更新 | Redis Pub/Sub |
| 群信息 | `group:{id}` | 5min | 变更后删除 |
| 部门树 | `dept:tree:{tenant_id}` | 10min | HR 同步后刷新 |

### 11.3 并发优化

| 策略 | 说明 |
|------|------|
| 无状态服务 | 所有业务服务无状态，K8s HPA 自动扩缩 |
| 连接池复用 | HTTP 长连接 + MongoDB/Redis 连接池 |
| 消息批量写入 | Kafka 生产者批量 flush，MongoDB bulk write |
| 异步非阻塞 | 审计日志、推送等非关键路径异步化 |
| 限流降级 | 超出容量返回 429，避免雪崩 |
